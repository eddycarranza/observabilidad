# Respuestas a las preguntas

# 1. ¿Por qué necesitamos Loki además de Prometheus si ya tenemos `/metrics`?

Prometheus y Loki resuelven problemas fundamentalmente distintos. Prometheus recolecta métricas: valores numéricos agregados en el tiempo (% CPU, requests/s, latencia p99). El endpoint `/metrics` expone esos números en formato scrape, pero no contiene texto narrativo de lo que ocurrió.

Loki almacena logs: líneas de texto sin estructura fija que describen eventos concretos (errores de autenticación, stack traces, mensajes de negocio como "pago rechazado"). Esa información textual no se puede modelar como una serie temporal numérica. Por eso ambas herramientas coexisten: Prometheus para "¿cuánto?" y Loki para "¿qué pasó exactamente?". En la práctica, cuando una alerta de CPU se dispara en Prometheus, se van a los logs de Loki para entender la causa raíz.

# 2. ¿Qué ventaja aporta que las fuentes de datos de Grafana estén aprovisionadas como código y no creadas a mano?



- Reproducibilidad: cualquier persona que haga `docker compose up` obtiene exactamente el mismo entorno, sin pasos manuales que se puedan olvidar o ejecutar de forma diferente.
- Versionado: los archivos de configuración viven en Git, por lo que se puede rastrear quién cambió qué y cuándo, y revertir si algo se rompe.
- Idempotencia: ejecutar el stack N veces produce siempre el mismo resultado; no hay "estado oculto" acumulado en una UI.
- Onboarding instantáneo: un nuevo integrante del equipo no necesita documentación de "cómo conectar Grafana a Prometheus"; simplemente levanta el stack.
- Auditoría y revisión: los cambios en configuración pasan por pull requests, igual que el código, lo que facilita la revisión y el control de calidad.

# 3. El panel "CPU contenedor" y el panel "CPU host" pueden mostrar valores muy distintos. ¿Por qué? ¿Cuál usarías para alertar sobre una aplicación concreta?

Por qué   el panel de CPU host (`node_cpu_seconds_total`) mide el uso total de todos los procesadores de la máquina física o VM, sumando todos los procesos: el SO, el daemon de Docker, los exporters, Grafana, Loki, y todas las apps. Es una vista agregada de toda la infraestructura.

El panel de CPU contenedor (`container_cpu_usage_seconds_total{name="lab-backend"}`) mide únicamente los ciclos de CPU consumidos por ese contenedor específico. Si el host tiene 4 cores al 20% de uso total pero el backend está al 80% de un core, el panel host mostraría ~20% y el de contenedor mostraría 80%.

¿Cuál usar para alertar sobre una aplicación concreta?  
El panel de CPU por contenedor. La alerta debe apuntar exactamente al servicio que se quiere monitorear, no al ruido de todo el host. Así se evitan falsas alarmas (el host sube por otro proceso) y se detectan problemas reales del backend aunque el host en conjunto esté tranquilo.

# 4. ¿Qué diferencia hay entre el evaluation interval y el pending period de una alarma?

-Evaluation interval (intervalo de evaluación): cada cuánto tiempo Grafana evalúa la condición de la regla de alerta. Con `10s`, Grafana ejecuta la consulta PromQL cada 10 segundos y comprueba si el resultado supera el umbral. Es la "frecuencia de muestreo" de la alarma.

-Pending period (período pendiente): cuánto tiempo seguido debe mantenerse la condición verdadera antes de que la alarma pase a estado `Firing`. Con `30s`, aunque la CPU supere el 50% en la primera evaluación, la alarma no se dispara de inmediato: debe permanecer por encima del umbral durante al menos 30 segundos consecutivos. Si cae antes de ese tiempo, vuelve a `Normal` sin haber disparado.

La distinción es importante: el *evaluation interval* controla la rapidez de detección; el pending period controla la tolerancia a picos transitorios. Un pending period corto detecta problemas rápido pero genera más falsas alarmas; uno largo es más conservador pero puede demorar la notificación ante incidentes reales.

# Nota sobre el entorno de ejecución (Windows + Docker Desktop)

El laboratorio fue ejecutado en Windows con Docker Desktop (backend WSL2). En este entorno, cAdvisor no expone el label `name` de los contenedores individuales, por lo que la query original del laboratorio:
 
 ACA PROFESOR TUVE UN ERROR AL MOMENTO DE USAR EL COMANDO SIGUIENTE  TUVE QUE USAR OTRO COMANDO PARA PODER FUNCIONAR 

```
sum(rate(container_cpu_usage_seconds_total{name="lab-backend"}[1m])) * 100
```

no retorna datos Esta es una limitación conocida de cAdvisor cuando se ejecuta sobre Docker Desktop en Windows: el daemon de Docker corre dentro de una VM interna (`docker-desktop`) y cAdvisor no puede acceder a los namespaces de los contenedores individuales de la misma forma que en Linux nativo.

Query equivalente utilizada en este entorno:

```
sum(rate(container_cpu_usage_seconds_total{id="/docker"}[1m])) * 100
```

Esta query muestra la CPU de todos los contenedores Docker agrupados, que es el equivalente funcional disponible en Docker Desktop. En un entorno Linux nativo (o Docker Engine dentro de Ubuntu WSL2 sin Docker Desktop), la query original del profesor funcionaría correctamente.

1. Ir a Dashboards y abrir el dashboard creado
2. Confirmar que existen los 4 paneles:

   - CPU del host (%) — muestra serie temporal del host
   - Logs de aplicación (API + frontend) — muestra logs en tipo Logs
   - Logs de infraestructura — muestra logs del stack

4. Confirmar que los paneles de logs muestran actividad reciente

# 6. Verificar la alarma de CPU > 50%

1. Ir a Alerting → Alert rules
2. Confirmar que existe la regla `CPU backend > 50%` en estado `Normal`
3. En http://localhost:8080 pulsar "Generar carga de CPU (30s)"
   (alternativa: `curl "http://localhost:3001/load?seconds=60"`)
4. Observar el panel de CPU del backend superar 50%
5. En Alerting → Alert rules, observar la transición: `Normal` → `Pending` → `Firing`
6. Esperar a que termine la carga y confirmar que vuelve a `Normal`

### 7. Verificar ciclo alarma → log (webhook)

1. En Grafana ir a Alerting → Contact points
2. Confirmar que existe un contact point de tipo Webhook apuntando a `http://backend:3001/alerts`
3. Generar carga nuevamente para disparar la alarma
4. En el panel "Logs de infraestructura" del dashboard, buscar el log de alerta recibido por el backend
